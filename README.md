## 第一部分：入门指南（适合所有人）

### 1. 什么是ScreenToGIF？

**简单来说**：ScreenToGIF 是一款能将你在电脑屏幕上的操作录制下来，并转换成GIF动图的工具。

**打个比方**：就像你用手机录制一段小视频，但最后得到的不是视频文件，而是一张会动的图片。这种动图可以在不点击播放的情况下自动展示内容，非常适合制作教程、展示操作步骤、分享有趣瞬间等。

**专业定义**：ScreenToGIF 是一款基于Python和PySide6开发的跨平台屏幕录制工具，能够捕获指定屏幕区域的画面，并以多线程方式高效处理图像数据，最终输出为优化后的GIF动画文件。软件支持实时预览、帧编辑、播放控制等高级功能。

**适用人群**：
- 👨‍🏫 **教师**：制作教学演示
- 👩‍💻 **程序员**：展示代码操作和Bug复现
- 🎨 **设计师**：展示设计过程和作品预览
- 📱 **社交媒体运营**：制作趣味动图
- 🎮 **游戏玩家**：分享游戏精彩瞬间
- 📝 **文档编写者**：制作操作说明文档

### 2. 快速开始（3分钟上手）

如果你是第一次使用，跟着以下步骤，3分钟内就能制作出你的第一个GIF：

**步骤1：启动软件**
- 双击 `ScreenToGIF.exe` 启动程序
- 你会看到一个分为左右两部分的窗口（左侧是控制面板，右侧是预览区域）

**步骤2：选择录制区域**
- 点击左侧的 **"选择录制区域"** 按钮（蓝色）
- 整个屏幕会变成半透明遮罩
- 按住鼠标左键，拖动选择你要录制的区域（比如一个浏览器窗口、一个软件界面）
- 松开鼠标，自动返回主窗口

**步骤3：开始录制**
- 点击 **"开始录制"** 按钮（绿色）
- 程序开始捕获你选择的区域
- 此时你可以进行想要录制的操作（比如点击菜单、输入文字等）

**步骤4：停止录制**
- 操作完成后，点击 **"停止录制"** 按钮（红色）
- 录制结束，右侧预览区会显示你录制的画面

**步骤5：保存GIF**
- 点击 **"生成GIF"** 按钮（蓝色）
- 选择保存位置和文件名
- 点击"保存"
- 等待进度条走完，你的GIF就制作完成啦！

**完成！** 🎉 现在去你保存的位置看看你的第一个GIF作品吧！

### 3. 软件安装与环境配置

#### 3.1 对于普通用户（无需编程知识）

**下载预编译版本**：
1. 访问项目GitHub页面：https://github.com/duma520/ScreenToGIF
2. 在"Releases"或"发布"页面找到最新版本
3. 下载 `ScreenToGIF-v1.0.4.zip` 或 `.exe` 安装包
4. 解压压缩包（如果是zip格式）或双击安装
5. 双击 `ScreenToGIF.exe` 即可运行

**系统要求**：
- Windows 7/8/10/11（64位）
- macOS 10.14+（需自行编译）
- Linux（需自行编译）
- 内存：至少2GB（推荐4GB以上）
- 硬盘空间：100MB以上

#### 3.2 对于开发者（从源码运行）

**环境要求**：
- Python 3.8 或更高版本
- pip（Python包管理器）
- Git（可选，用于克隆代码）

**安装步骤**：

```bash
# 1. 克隆代码库
git clone https://github.com/duma520/ScreenToGIF.git
cd ScreenToGIF

# 2. 创建虚拟环境（推荐）
python -m venv venv

# 3. 激活虚拟环境
# Windows:
venv\Scripts\activate
# macOS/Linux:
source venv/bin/activate

# 4. 安装依赖
pip install -r requirements.txt

# 5. 运行程序
python ScreenToGIF.py
```

**requirements.txt 内容**：
```
PySide6>=6.4.0
opencv-python>=4.8.0
numpy>=1.24.0
Pillow>=10.0.0
```

#### 3.3 使用打包工具（高级）

项目提供了多个批处理文件用于打包：

- `0_nuitka_pyside6_msvc.bat`：使用Nuitka打包（生成更小的可执行文件）
- `0_pyinstaller.bat`：使用PyInstaller打包（更快的打包速度）
- `9_spec.bat`：使用spec文件打包（自定义配置）

### 4. 界面介绍

![界面布局示意图]

ScreenToGIF的主界面分为以下几个主要区域：

#### 4.1 左侧控制面板

**① 录制控制组**
- **选择录制区域**（蓝色按钮）：点击后进入区域选择模式
- **开始录制**（绿色按钮）：开始捕获屏幕
- **停止录制**（红色按钮）：结束录制
- **区域信息**：显示当前选择的区域尺寸（如"区域: 800 × 600"）

**② 录制设置组**
- **帧率 (FPS)**：设置每秒捕获的帧数（1-30）
- **缩放比例**：输出GIF的尺寸百分比（10-200%）
- **GIF质量**：压缩质量设置（1-100%）
- **颜色数**：使用的颜色数量（2-256色）
- **自动开始录制**：勾选后，选择区域自动开始录制
- **显示实时预览**：勾选后，录制时右侧会实时显示画面

**③ 输出设置组**
- **输出目录**：显示GIF文件保存的文件夹路径
- **打开文件夹**：快速打开输出目录

**④ 录制后操作组**
- **播放录制**（青色）：预览录制的帧序列
- **编辑帧**（黄色）：进入帧编辑器（删除、调整顺序）
- **生成GIF**（蓝色）：将录制的帧保存为GIF文件

**⑤ 处理进度组**
- **状态标签**：显示当前操作状态
- **进度条**：显示GIF生成进度

#### 4.2 右侧预览面板

**⑥ 实时预览区**
- 显示当前录制的画面（仅录制时有效）
- 可缩放适应窗口大小

**⑦ 帧信息**
- 显示已捕获的帧数（如"已捕获: 150 帧"）

**⑧ 录制历史**
- 显示最近生成的GIF文件列表
- 双击可打开文件所在位置

#### 4.3 菜单栏

- **文件**：打开输出文件夹、退出程序
- **编辑**：清除所有帧
- **视图**：窗口置顶切换
- **帮助**：关于信息

#### 4.4 状态栏

- 显示当前操作状态信息

### 5. 如何选择录制区域

#### 5.1 基本选择方法

1. 点击主界面的 **"选择录制区域"** 按钮
2. 屏幕变为半透明遮罩，鼠标变为十字光标
3. **按住左键**并拖动，框选要录制的区域
4. 松开鼠标，自动返回主窗口

**技巧**：
- 框选时，角落会显示当前尺寸（如"800 × 600"）
- 如果选错了，按 **ESC键** 取消选择

#### 5.2 精确选择技巧

**场景1：录制整个窗口**
- 打开目标窗口（如浏览器）
- 进入选择模式后，从窗口左上角拉到右下角
- 或者使用窗口截图工具先测量尺寸

**场景2：录制固定区域（如软件工具栏）**
- 测量目标区域的大致尺寸
- 在选择时留意角落显示的尺寸信息
- 可以多次尝试直到选中精确区域

**场景3：录制多个屏幕**
- 程序支持多显示器
- 选择区域可以跨越多个屏幕
- 所有屏幕会合并为一个虚拟桌面

#### 5.3 保存上次选择

程序会自动保存你上次选择的区域。下次启动时：
- 区域信息会显示上次的尺寸
- "开始录制"按钮会自动启用
- 点击即可开始录制相同区域

### 6. 录制你的第一个GIF

#### 6.1 基础录制流程

**场景示例：录制一个简单的菜单操作**

```
目标：录制"打开记事本→输入文字→保存文件"的过程
```

**步骤详解**：

1. **准备阶段**
   - 打开ScreenToGIF
   - 点击"选择录制区域"
   - 框选包含"开始菜单"和记事本窗口的区域
   - 确认区域信息显示正确

2. **开始录制**
   - 点击"开始录制"（绿色按钮）
   - 注意右上角开始计时

3. **执行操作**
   - 点击"开始菜单"
   - 输入"记事本"并打开
   - 在记事本中输入"Hello World"
   - 点击"文件→保存"
   - 选择保存位置（这个过程也会被录制）

4. **停止录制**
   - 点击"停止录制"（红色按钮）
   - 右侧预览区显示刚才的操作

5. **保存GIF**
   - 点击"生成GIF"
   - 选择文件名（如"记事本教程.gif"）
   - 点击"保存"
   - 等待进度条完成

**完成！** 现在你有了一个完整的操作教程GIF。

#### 6.2 录制注意事项

| 注意事项 | 说明 | 建议 |
|---------|------|------|
| **录制时长** | 录制时间越长，帧数越多，GIF文件越大 | 控制在30秒内最佳 |
| **操作速度** | 操作太快可能导致关键步骤丢失 | 放慢操作，每一步停顿1-2秒 |
| **鼠标移动** | 鼠标轨迹也会被录制 | 尽量保持鼠标稳定 |
| **窗口变化** | 如果窗口移动出录制区域，内容会丢失 | 固定窗口位置 |

#### 6.3 常见录制场景示例

**场景1：软件操作教程**

```
- 缩放: 100%（保持清晰度）
```

**场景2：游戏精彩瞬间**
```
适合：游戏分享、攻略
```

**场景3：UI/UX演示**
```
适合：设计师展示交互

- FPS: 12-15（展示过渡动画）
- 缩放: 100%（原始尺寸）

```

### 7. 保存与导出设置

#### 7.1 保存位置

程序默认将GIF文件保存在：
```
程序所在目录\ScreenToGIF_Recordings\
```

例如：
- 如果程序在 `D:\Tools\ScreenToGIF\`
- 输出目录为 `D:\Tools\ScreenToGIF\ScreenToGIF_Recordings\`

**为什么这样设计？**
- 便于管理：所有录制文件集中存放
- 便于携带：移动整个文件夹时，录制文件一起移动
- 避免权限问题：不需要管理员权限即可写入

#### 7.2 文件命名规则

默认文件名格式：
```
screen_recording_YYYYMMDD_HHMMSS.gif
```

示例：
- `screen_recording_20260311_143025.gif`（2026年3月11日 14:30:25）

**建议的自定义命名方式**：
- `软件名_功能_日期.gif`（如：`photoshop_抠图教程_20260311.gif`）
- `项目_版本_操作.gif`（如：`app_v1.2_登录流程.gif`）

#### 7.3 导出设置详解

**帧率 (FPS) 设置建议**：

| 使用场景 | 推荐FPS | 理由 |
|---------|---------|------|
| 简单的界面截图（点击按钮） | 5-8 | 足够捕捉关键动作，文件小 |
| 软件操作教程 | 8-12 | 平衡流畅度和文件大小 |
| UI动画演示 | 12-15 | 捕捉过渡动画细节 |
| 快速动作（游戏、鼠标移动） | 15-20 | 保证动作流畅 |
| 极限情况 | 20-30 | 文件极大，慎用 |

**缩放比例设置建议**：

| 使用场景 | 推荐缩放 | 理由 |
|---------|---------|------|
| 全高清录制（1920×1080） | 50-70% | 减小文件大小，同时保持可读性 |
| 细节展示（小图标、文字） | 100-150% | 保持细节清晰 |
| 长文档滚动 | 80-100% | 平衡可视范围和文件大小 |
| 手机屏幕录制 | 100% | 原始尺寸最合适 |

**颜色数设置建议**：

| 颜色数 | 适用场景 | 文件大小 | 画质 |
|-------|---------|---------|------|
| 256色 | 色彩丰富的界面、游戏 | 大 | 极好 |
| 128色 | 普通软件界面 | 中 | 好 |
| 64色 | 简单界面、文字教程 | 小 | 一般 |
| 32色 | 图标、简单图形 | 很小 | 可接受 |
| 16色以下 | 极简黑白内容 | 极小 | 差 |

**质量设置建议**：

| 质量 | 适用场景 | 文件大小 | 说明 |
|-----|---------|---------|------|
| 90-100% | 最终成品、重要内容 | 最大 | 几乎无损 |
| 70-90% | 普通分享 | 大 | 视觉无损 |
| 50-70% | 网络发布、邮件附件 | 中 | 可接受压缩 |
| 30-50% | 预览、草稿 | 小 | 明显压缩 |
| 30%以下 | 极简需求 | 极小 | 画质损失严重 |

#### 7.4 批量导出

目前版本不支持批量导出，但你可以：
1. 多次录制，分别保存
2. 使用外部工具合并GIF（如：FFmpeg、GIMP）

### 8. 常见问题解决

#### 8.1 程序无法启动

**问题现象**：双击程序没反应，或闪退

**解决方法**：

1. **检查Python环境**（源码运行）：
```bash
python --version  # 确保是3.8+
pip list | findstr PySide6  # 检查依赖
```

2. **检查依赖库**：
```bash
pip install -r requirements.txt --upgrade
```

3. **查看错误日志**：
- 在命令行中运行程序，查看错误信息
- 错误信息通常会指出缺失的模块

4. **管理员权限运行**：
- 右键点击程序，选择"以管理员身份运行"

#### 8.2 录制区域选择问题

**问题1：选择区域时看不到遮罩**
- 确保没有其他全屏程序覆盖
- 尝试按 `Win+D` 显示桌面后再试

**问题2：选择的区域不是我想要的**
- 按 `ESC` 取消选择，重新框选
- 注意鼠标拖动时要精确

**问题3：无法选择多显示器区域**
- 程序自动支持多显示器
- 确保显示器设置为"扩展"模式

#### 8.3 录制过程中卡顿

**问题现象**：录制时画面不流畅，鼠标移动卡顿

**原因和解决方法**：

| 原因 | 解决方法 |
|-----|---------|
| FPS设置过高 | 降低FPS到8-10 |
| 录制区域太大 | 缩小录制区域，或降低缩放比例 |
| 电脑性能不足 | 关闭其他程序，释放CPU/内存 |
| 磁盘写入慢 | 更换保存位置到SSD |

#### 8.4 生成的GIF文件太大

**优化方法**：

1. **降低FPS**：从15降到8，文件大小可减少40%
2. **减少颜色数**：从256降到128，文件大小可减少30%
3. **降低缩放**：从100%降到80%，文件大小可减少36%
4. **缩短录制时间**：尽量控制在15秒内
5. **降低质量**：从90%降到70%，文件大小可减少25%

**组合优化示例**：

```
原始：FPS 15, 颜色256, 缩放100%, 质量90% → 文件 5.2MB

优化后：FPS 10, 颜色128, 缩放80%, 质量70% → 文件 1.8MB (减小65%)
```

#### 8.5 GIF颜色失真

**问题现象**：GIF中的颜色和原始屏幕不一样

**原因**：
- GIF格式限制最多256色
- 颜色量化过程会丢失部分颜色

**解决方法**：
- 保持颜色数为256（最大）
- 对于渐变色多的内容，考虑使用视频格式
- 或者使用其他工具转成视频（如MP4）

#### 8.6 保存时提示权限错误

**问题现象**：无法保存文件，提示权限不足

**解决方法**：
1. 检查输出目录是否存在，是否有写入权限
2. 尝试以管理员身份运行程序
3. 更改保存位置到用户文件夹（如 `C:\Users\用户名\Videos\`）

### 9. 帧率(FPS)详解与选择建议

#### 9.1 什么是帧率？

**通俗解释**：
帧率（FPS，Frames Per Second）就是每秒钟捕获多少张图片。就像翻书动画一样，图片越多，动画越流畅。

**类比**：
- 5 FPS：每秒5张图 → 像幻灯片，卡顿感明显
- 10 FPS：每秒10张图 → 基本流畅，适合教程
- 15 FPS：每秒15张图 → 很流畅，适合动画
- 24 FPS：每秒24张图 → 电影级流畅
- 30 FPS：每秒30张图 → 超高流畅，文件巨大

#### 9.2 帧率对GIF的影响

**视觉流畅度**：
```
卡顿 ←──────────→ 流畅
5fps   10fps   15fps   24fps   30fps
```

**文件大小**（相同内容下）：
- 10秒录制，100%缩放，256色

| FPS | 帧数 | 估算文件大小 | 流畅度 |
|----|------|------------|--------|
| 5 | 50帧 | 1.5MB | 卡顿，适合极简操作 |
| 8 | 80帧 | 2.4MB | 可接受，适合教程 |
| 10 | 100帧 | 3.0MB | 基本流畅，推荐 |
| 12 | 120帧 | 3.6MB | 流畅，适合UI演示 |
| 15 | 150帧 | 4.5MB | 很流畅，适合动画 |
| 20 | 200帧 | 6.0MB | 非常流畅，文件大 |
| 30 | 300帧 | 9.0MB | 极流畅，文件巨大 |

#### 9.3 不同场景的FPS选择指南

**场景1：软件操作教程**
```
目标：展示点击按钮、输入文字等操作
推荐FPS：8-10
理由：

- 文件大小适中，便于分享

```

**场景2：UI交互动画**
```
目标：展示菜单展开、页面切换等动画效果
推荐FPS：12-15
理由：
- 动画需要流畅才能体现设计
- 15fps足以捕捉大部分UI动画
- 文件大小仍在可接受范围

示例：展示手机App的滑动效果
```

**场景3：游戏精彩瞬间**
```
目标：捕捉快速动作（跳跃、射击、技能释放）
推荐FPS：15-20

- 游戏动作快，需要高FPS捕捉细节

- 权衡画质和文件大小
```

### 10. 缩放比例的应用场景

#### 10.1 什么是缩放比例？

**通俗解释**：
缩放比例就是输出GIF的尺寸是原始录制区域的百分之多少。

- 100% = 原始大小
- 50% = 宽度和高度都缩小一半
- 200% = 宽度和高度都放大一倍

**计算公式**：
```
输出宽度 = 录制区域宽度 × (缩放比例 ÷ 100)
输出高度 = 录制区域高度 × (缩放比例 ÷ 100)
```

**示例**：
录制区域 1920×1080
- 缩放50% → 输出 960×540
- 缩放100% → 输出 1920×1080
- 缩放150% → 输出 2880×1620

#### 10.2 缩放对文件大小的影响

文件大小与像素数量成正比。像素数量变化公式：
```
新像素数 = 原像素数 × (缩放比例 ÷ 100)²
```

**示例**（1920×1080 = 约207万像素）：

| 缩放比例 | 输出尺寸 | 像素数 | 文件大小估算 |
|---------|---------|-------|------------|
| 50% | 960×540 | 约52万 | 25%（减少75%） |
| 70% | 1344×756 | 约102万 | 49%（减少51%） |
| 80% | 1536×864 | 约133万 | 64%（减少36%） |
| 90% | 1728×972 | 约168万 | 81%（减少19%） |
| 100% | 1920×1080 | 约207万 | 100%（基准） |
| 120% | 2304×1296 | 约299万 | 144%（增加44%） |
| 150% | 2880×1620 | 约466万 | 225%（增加125%） |

#### 10.3 不同场景的缩放选择

**场景1：社交媒体分享**
```
平台限制：Twitter最大5MB，微信限制2-5MB
推荐缩放：50-70%
理由：
- 手机屏幕小，大尺寸浪费
- 减小文件便于上传
- 文字仍可辨认

示例：1920×1080 → 缩放60% → 1152×648
```

**场景2：技术文档嵌入**
```
需求：文字清晰，图标可辨
推荐缩放：80-100%
理由：
- 文档可能被放大查看
- 代码和文字需要清晰
- 文件大小不是首要考虑

示例：IDE操作教程，保持文字可读
```

**场景3：展示精细设计**
```
需求：展示UI细节、图标细节
推荐缩放：100-150%
理由：
- 放大可以看清像素级细节
- 设计师需要展示精确效果
- 文件大小可以接受

示例：展示App图标设计过程
```

**场景4：邮件附件**
```
限制：多数邮件系统限制附件10-20MB
推荐缩放：50-70%
理由：
- 邮件附件需要控制大小
- 收件人可能不会放大查看
- 确保能被正常接收

示例：发送操作教程给客户
```

#### 10.4 缩放与文字可读性

**重要原则**：缩放后文字必须保持可读

**测试标准**：
- 中文字体：至少12px
- 英文字体：至少10px

**示例**：
```
录制区域：2560×1440（高分屏）
- 系统字体通常较大
- 缩放50%后：1280×720
- 字体大小也缩小50%
- 需要检查文字是否可读

建议：
- 高分屏录制时，缩放不低于70%
- 如果内容包含小字，保持80%以上
```

#### 10.5 智能缩放技巧

**技巧1：只缩放不必要的区域**
- 录制时包含操作区域和周边空白
- 通过缩放裁剪掉空白区域
- 既减小文件，又保持核心区域清晰

**技巧2：针对重点区域放大**
- 录制整体操作，但在编辑时裁剪重点
- 或者分两次录制：整体和细节

**技巧3：使用固定输出尺寸**
- 如统一输出 800×600
- 根据录制区域计算最佳缩放
- 保持所有教程统一尺寸

### 11. 颜色数与GIF文件大小的关系

#### 11.1 GIF颜色原理

**通俗解释**：
GIF是一种古老的图片格式，最多只能存储256种颜色。如果原始画面颜色超过256种，就需要通过"颜色量化"算法选择最重要的256种颜色。

**颜色量化过程**：
1. 分析每一帧的所有颜色
2. 找出出现频率最高的颜色
3. 创建包含这些颜色的调色板
4. 用调色板中的颜色近似表示其他颜色

#### 11.2 颜色数对文件大小的影响

**理论关系**：
- 颜色数越少，调色板越小
- 每个像素需要的位数越少
- 文件大小相应减小

**实际测试数据**（10秒录制，10fps，800×600）：

| 颜色数 | 调色板大小 | 文件大小 | 画质 | 适用场景 |
|-------|-----------|---------|------|---------|
| 256色 | 768字节 | 4.2MB | 最佳 | 照片级内容 |
| 128色 | 384字节 | 3.6MB | 优秀 | 彩色界面 |
| 64色 | 192字节 | 2.9MB | 良好 | 普通软件 |
| 32色 | 96字节 | 2.3MB | 一般 | 图标为主 |
| 16色 | 48字节 | 1.8MB | 较差 | 黑白内容 |
| 8色 | 24字节 | 1.4MB | 很差 | 极简图形 |
| 4色 | 12字节 | 1.1MB | 极差 | 特殊用途 |

#### 11.3 不同内容的颜色数选择

**场景1：照片/游戏画面**
```
内容特点：色彩丰富，渐变多
推荐颜色：256色
理由：
- 需要尽可能保留原色
- 减少颜色数会导致色块
- 画质优先

示例：游戏精彩片段、摄影教程
```

**场景2：软件界面**
```
内容特点：固定配色，颜色有限
推荐颜色：128色或64色
理由：
- 大多数软件界面颜色少于64色
- 128色足够保留所有界面颜色
- 可以减小文件大小

示例：Windows设置界面、Word操作
```

**场景3：代码编辑器**
```
内容特点：背景单一，语法高亮
推荐颜色：32色或16色
理由：
- 编辑器通常只有几种颜色
- 语法高亮一般不超过10种颜色
- 可以用很少颜色完美还原

示例：VS Code编写代码过程
```

**场景4：黑白/灰度内容**
```
内容特点：只有黑白灰
推荐颜色：8色或4色
理由：
- 灰度可以用很少的色阶表示
- 文件极小，画质影响小
- 适合打印文档演示

示例：终端命令行操作
```

#### 11.4 颜色数对画质的影响

**肉眼观察效果**：

| 颜色数 | 效果描述 |
|-------|---------|
| 256色 | 基本看不出损失，渐变可能有轻微条纹 |
| 128色 | 仔细看能发现颜色过渡不平滑 |
| 64色 | 明显色块，尤其是渐变区域 |
| 32色 | 严重色块，颜色失真明显 |
| 16色 | 卡通化效果，颜色跳跃 |

**测试方法**：
1. 用不同颜色数生成同一个GIF
2. 在屏幕上同时对比
3. 选择"刚好满意"的最低颜色数

#### 11.5 颜色优化的高级技巧

**技巧1：自适应调色板**
程序使用 `Image.ADAPTIVE` 模式，会为每帧优化调色板。这意味着：
- 每帧可以使用不同的颜色集
- 更准确地表现每帧的内容
- 但会增加处理时间

**技巧2：抖动效果**（当前版本未实现）
- 通过像素排列模拟更多颜色
- 视觉效果更好
- 但文件会略微增大

**技巧3：颜色限制策略**
```
如果画面变化不大：
- 可以使用全局调色板
- 颜色数可以减少到32色
- 文件可减小40%

如果画面变化剧烈：
- 必须使用每帧调色板
- 颜色数保持在128以上
- 文件可能增大
```

### 12. 录制后编辑功能详解

#### 12.1 什么是帧编辑器？

帧编辑器是ScreenToGIF的核心功能之一，允许你在生成GIF之前，对录制的帧进行修改和调整。

**可以做什么**：
- ✅ 删除不需要的帧
- ✅ 调整帧的顺序
- ✅ 批量删除选定帧
- ✅ 预览播放效果
- ✅ 重置到原始状态

**不能做什么**（当前版本）：
- ❌ 裁剪单帧画面
- ❌ 添加文字/标注
- ❌ 调整单帧亮度/对比度
- ❌ 合并多个录制

#### 12.2 如何进入帧编辑器

1. 录制完成后（或打开已有录制）

- **播放/暂停**：预览编辑效果

**② 左侧帧列表**

- 每帧有编号（帧1, 帧2...）
- 可以单选或多选（按住Ctrl）

**③ 右侧预览区**
- 显示当前选中帧的大图
- 滑块可快速跳转
- 显示当前帧/总帧数

**④ 底部按钮**

- **取消**：放弃所有编辑

### 15. 多线程处理机制

#### 15.1 为什么需要多线程？

**单线程的问题**：
```
假设录制30秒，每秒10帧，共300帧
编码需要5秒
如果是单线程：
- UI会卡死5秒
- 用户无法进行任何操作
- 体验极差
```

**多线程的解决方案**：
```
主线程：UI界面，响应用户操作
捕获线程：专门负责抓取屏幕
编码线程：专门负责生成GIF
三个线程并行工作，互不阻塞
```

#### 15.2 线程模型图

```
时间线 →
─────────────────────────────────────
主线程：显示界面 ←→ 处理用户点击 ←→ 更新进度条
        │              │              ↑
        │ 启动          │ 停止         │ 信号
        ▼              ▼              │
捕获线程：──捕获1──捕获2──...──停止────┘
                                         │
                                         启动
                                         ▼
编码线程：                          ──处理1──处理2──...
```

#### 15.3 线程通信机制

**使用Qt信号/槽**：
```python
# 定义信号
class ScreenCaptureThread(QThread):
    frame_captured = Signal(np.ndarray)  # 传递帧数据
    capture_finished = Signal()           # 完成信号
    capture_error = Signal(str)           # 错误信号

# 连接信号
self.capture_thread.frame_captured.connect(self.on_frame_captured)
self.capture_thread.capture_error.connect(self.on_capture_error)

# 发射信号
self.frame_captured.emit(frame)
```

**使用Event控制线程**：
```python
# 线程控制
self.capture_event = Event()
self.capture_event.set()  # 开始

# 线程循环中检查
while self.is_running:
    if not self.capture_event.is_set():
        break  # 退出循环
    # ... 捕获逻辑
```

#### 15.4 线程安全的数据共享

**原则**：不要直接共享可变数据

**正确做法**：
```python
# 在捕获线程中收集帧
self.frames.append(frame)  # 线程内部列表

# 主线程获取副本
frames_copy = self.capture_thread.get_frames()  # 返回.copy()

# 传递给编码线程时也使用副本
self.encode_thread = GIFEncodingThread(
    self.captured_frames.copy(),  # 关键：使用副本
    # ... 其他参数
)
```

**错误做法**：
```python
# 不要这样做！
self.captured_frames = self.capture_thread.frames  # 直接引用
# 两个线程会同时修改同一个列表
```

#### 15.5 线程生命周期管理

**启动顺序**：
```python
def _on_start_capture(self):
    # 1. 重置状态
    self.captured_frames = []
    
    # 2. 创建线程
    self.capture_thread = ScreenCaptureThread(self.capture_rect)
    
    # 3. 连接信号
    self.capture_thread.frame_captured.connect(...)
    
    # 4. 启动线程
    self.capture_thread.start()
```

**停止顺序**：
```python
def _on_stop_capture(self):
    # 1. 设置停止标志
    self.is_recording = False
    
    # 2. 通知线程停止
    if self.capture_thread:
        self.capture_thread.stop()  # 设置内部标志
        
    # 3. 等待线程结束（带超时）
    self.capture_thread.wait(1000)  # 最多等1秒
    
    # 4. 清理线程对象
    self.capture_thread = None
```

#### 15.6 线程性能考虑

**CPU使用**：
- 捕获线程：约5-15% CPU（取决于FPS和区域大小）
- 编码线程：约20-50% CPU（取决于帧数和缩放）
- 总使用：25-65%，对现代CPU友好

**内存使用**：
- 每帧约 1920×1080×3 = 6.2MB
- 10秒录制（100帧）≈ 620MB
- 程序会占用较多内存，建议：
  - 不要录制过长
  - 及时生成GIF释放内存
  - 4GB内存电脑建议录制不超过5秒

**最佳实践**：
```python
# 帧率与内存关系
# 10fps × 30秒 = 300帧 × 6.2MB = 1.86GB
# 建议：15秒内录制，或降低FPS
```

#### 15.7 错误处理与恢复

**超时处理**：
```python
def wait_with_timeout(self, thread, timeout_ms):
    """等待线程结束，带超时"""
    if not thread.wait(timeout_ms):
        print("线程未正常结束，强制终止")
        thread.terminate()  # 不推荐，仅作最后手段
        thread.wait()
```

**异常传播**：
```python
try:
    # 可能出错的捕获逻辑
    frame = cv2.cvtColor(...)
except Exception as e:
    # 通过信号传递错误
    self.capture_error.emit(f"捕获失败: {str(e)}")
    # 继续运行，不要崩溃
```

### 29. 术语解释

#### A

**APNG**：Animated Portable Network Graphics，动态PNG，支持24位真彩色和8位透明，比GIF画质更好但兼容性略差。

#### B

**BGR**：Blue-Green-Red，OpenCV默认的颜色通道顺序，与常见的RGB顺序相反。

#### C

**CV2**：OpenCV的Python模块，用于图像处理和计算机视觉任务。

#### D

**dataclass**：Python的数据类装饰器，简化类的定义，自动生成__init__等方法。

#### F

**FPS**：Frames Per Second，每秒帧数，衡量动画流畅度的单位。

#### G

**GIF**：Graphics Interchange Format，图形交换格式，支持256色和简单动画。

#### I

**ICC**：International Color Consortium，国际色彩联盟，定义色彩管理标准。

#### J

**JSON**：JavaScript Object Notation，轻量级数据交换格式，用于存储配置。

#### L

**loop**：循环，GIF可以设置循环次数（0表示无限循环）。

#### M

**MVC**：Model-View-Controller，模型-视图-控制器，软件设计模式。

#### N

**NumPy**：Python的科学计算库，提供多维数组对象。

#### O

**OpenCV**：Open Source Computer Vision Library，开源计算机视觉库。

#### P

**PIL**：Python Imaging Library，Python图像处理库，Pillow是其升级版。

**PySide6**：Qt for Python，Qt6的Python绑定，用于创建GUI应用。

#### Q

**QThread**：Qt的线程类，用于实现多线程。

**QImage**：Qt的图像类，支持多种像素格式。

**QPixmap**：Qt的像素图类，优化用于屏幕显示。

#### R

**RGB**：Red-Green-Blue，红绿蓝三原色，常见的颜色模型。

#### S

**Signal/Slot**：Qt的信号/槽机制，用于对象间通信。

#### T

**Thread**：线程，程序执行的最小单元。

#### U

**UI**：User Interface，用户界面。

**UX**：User Experience，用户体验。

#### V

**Virtual Desktop**：虚拟桌面，多显示器组合成的逻辑桌面。

#### W

**WebP**：Google开发的图像格式，支持有损/无损压缩和动画。

#### 30.3 常见问题自助排查

**Q: 程序启动后闪退**
A: 
1. 检查Python版本（需3.8+）
2. 检查依赖库是否完整
3. 删除配置文件重启
4. 查看系统事件日志

**Q: 录制画面黑屏**
A:
1. 检查是否有程序在全屏运行
2. 尝试降低录制区域
3. 检查显卡驱动
4. 重启程序

**Q: GIF生成特别慢**
A:
1. 降低FPS
2. 降低缩放比例
3. 减少录制时长
4. 关闭其他程序
